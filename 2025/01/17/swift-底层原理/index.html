swift-底层原理

发表于 2025-01-17 更新于 2025-09-05 分类于 swift
本文主要介绍 swift的底层原理。

内存管理

Swift 对象的内存结构 HeapObject (OC objc_object) ，有两个属性： 一个是 Metadata ，一个是 RefCount ，默认占用 16 字节大小

Metadata包含如下：

  • 类型的基础信息（如类、结构体、协议）。
  • 方法和属性的布局。
  • 协议符合性。
  • 内存布局。
  • 动态分发所需的信息。

举例

  • 0x100008140 是指向对象元数据的指针(Metadata)。对象的元数据包含对超类元数据的引用、属性和方法列表以及几个私有字段。指向对象元数据的指针也称为isa 指针
  • 0x3就是引用计数(RefCount)
  • 0xffee 和 0x1 是 CommitInfo 对象的两个属性值

    强指针的引用计数是从第33位开始的，获取引用计数的计算方式同无主引用

  • weak 弱引用

    • 当对象有一个弱引用时，就会 动态创建 side table 管理内存。side table包含了原始对象的指针以及强引用、弱引用和无主引用的数量

    • 创建 side table 后，指向 side table 的指针会存储在 HeapObject 第8-16个字节中，也就是原来的引用计数器(RefCount)位置中

    • UserSlowRC 就是是否有 side table 的标记

      此后将设置 UseSlowRC 标志：

      数据结构如下

    • side table

      • nativeValue 存储指向 Swift 原生对象的弱引用。
      • nonnativeValue 表示指向 Objective-C 对象的弱引用。
      • 如果对象是 Swift 的原生对象，使用 nativeValue；如果是 Objective-C 对象，使用 nonnativeValue

    • 工作机制：

      • 当一个弱引用变量被声明时，Swift 运行时会为其分配一个 WeakReference 对象。
      • 如果对象还没有侧表，运行时会为其分配一个新的侧表（HeapObjectSideTableEntry），并在侧表中初始化弱引用计数。
      • 弱引用的使用如下：
        • 弱引用通过 WeakReference 对象间接访问原始对象
        • 1.如果对象仍然存活，则返回 HeapObject 的指针。
        • 2.如果对象已被销毁，则返回 nil。

OC:

  • 弱引用计数是存放在全局维护的散列表中，isa中会记录是否使用了散列表。
  • 在引用计数为0时，自动触发dealloc，会检查并清空当前对象的散列表计数。

Swift

  • 弱引用计数也是存放在散列表中，这个散列表是局部的! 每个weak对象都有自己的散列表!
  • 即便一个对象在使用了弱引用后，也不能保证相关内存全部被释放! 因为SideTable 的生命周期与对象是分离的，当强引用计数为 0 时，只有 HeapObject 被释放了,但不会触发SideTable的清空。而是在下次访问时发现当前对象为nil时，才会清空SideTable
  • Swift 对象生命周期 对象不立即销毁，而是经过 live -> deiniting -> deinited -> freed -> dead 这五个阶段
  • 当 strong 为0时 对象进入 deinited 状态；Unowned 访问触发断言；Weak 访问返回 nil

消息派发

方法调度分为两大类

  • 静态调度，速度快，但不灵活
  • 动态调度，速度较慢，但更灵活

更细分

  • 内联（最快，不灵活）：代码就在那里不需要寻找，就能调用
  • 静态调度：根据地址找到函数调用
  • 表调度：根据地址找到函数表，然后找到函数，调用
  • 消息分发（最慢，极其灵活）：是Objc的运行时系统，所有方法都是通过objc_msgSend或者super_msgSend来找到具体的函数进行调用

内联
这是最快的方法，实际上根本不是调度。内联是一种编译器优化，它实际上是用该函数中的代码替换该函数的调用。代码就放在那里直接使用


静态分发

  • 函数的内存位置在编译时是已知的
  • static 函数、值类型（例如 struct 和 enum）的方法。

优点

  • 确定性：编译器知道每个方法的具体实现和调用位置。
  • 性能优化：编译器可以对静态分发的函数进行内联优化或预计算。

动态分发

  1. 函数的具体实现地址并不是在编译时确定的，而是通过一张表（函数指针表）在运行时选择正确的实现。
  2. 支持多态性：一个基类的多个子类可以有不同的实现
  3. 只有类（class）可以使用虚拟表调用函数，结构体使用静态分发。
  4. 如果类被标记为 final，则虚拟表不会用于该类，因为它不能被继承，编译器可以进行静态分发优化。
    Swift 中的两种表分发机制
  • 虚表分发
    • 子类会继承父类的虚表，子类覆写会更新虚表对应方法的地址。
    • 运行时调用方法时，会根据对象的元数据找到函数虚表，找到函数的指针地址，根据指针地址跳转到方法在内存中的实现位置来调用方
      1. 跳转到虚拟表
      • 每个对象实例都包含一个指向所属类的元数据（metadata）的指针。
      • 元数据中包含该类的虚拟表地址。
      1. 查找方法地址
      • 在虚拟表中找到对应方法的指针地址。
      1. 跳转到方法实现
      • 根据指针地址跳转到方法在内存中的实现位置。
  • 协议表分发
    Existential Container 是 协议的底层实现

    • Value Buffer 存储值的内存区域。对于小型值（一般 ≤ 3 个字大小，如 3 个 64 位字），直接内嵌在容器中。如果值较大，则使用堆内存存储，容器保存一个指向堆内存的指针

    • Protocol Witness Table (协议表) 存储协议符合性信息。这是协议的动态分发机制的核心部分，包含方法实现的地址列表，类似虚表（vtable）

    • Metadata， 描述值的类型信息。例如，对于结构体 Dog，会包含它的类型元数据。

      当我们将 Dog 赋值给 Animal 类型的变量时，Swift 将创建一个 Existential Container 来存储 Dog 的实例及其协议的相关信息。
      底层运行时的步骤如下：

      1. 从 Existential Container 中取出 Protocol Witness Table (PWT)。
      2. 根据协议的方法索引查找对应方法的函数指针,调用实际的方法实现。

闭包

闭包是独立的功能块，可以在代码中传递和使用
闭包表达式

尾随闭包

尾随闭包

逃逸闭包：指闭包被传递到函数之外的作用域，或者在函数执行完之后才被调用的闭包

自动闭包：自动创建一个闭包用来包裹一个表达式，这种闭包不接受任何参数，当闭包被调用时，返回包裹在闭包中的表达式的值

内存结构

  • 没有捕获变量
    • 存储的就是一个函数指针，
    • 伪代码模拟数据结构
    • 针对没有捕获变量的闭包，为了避免内存开销，编译器对其进行优化了，实际就是函数调用，上面的数据结构是猜测的优化前的存储形式。
  • 捕获变量
    • 闭包会被存储在堆区
    • 伪代码模拟数据结构
  • 总结
    • 闭包的数据结构，函数指针，捕获列表、元数据组成
    • 捕获变量的布局由 Swift 编译器在编译期决定，并写入元数据中。元数据包括捕获变量的类型、大小和对齐信息。
    • 捕获列表存储闭包捕获的变量
    • 隐式捕获：系统会在堆区开辟空间来存储捕获的变量，如果是值类型捕获的是地址。可以理解为捕获的变量都被包装给了对象，存放在捕获列表中，是对象就有isa指针，能找到内存中的位置取出值来使用。有引用计数来管理内存，有元数据信息。
    • 显示捕获：对值进行copy

总结：

  1. 闭包是一个引用类型;
  2. 闭包捕获全局(静态)变量/常量默认是不捕获的，直接使用该变量/常量地址；
  3. 闭包捕获局部值类型变量会在堆区开辟内存空间；每次修改捕获变量的值的时候，其实是修改堆区当中的值；
  4. 闭包捕获局部引用类型变量，无需开辟堆区内存空间，直接可引用堆区地址；并且该变量引用计数+1，在闭包生命周期结束会把该变量引用计数-1。

思考

  1. 逃逸闭包不能捕获 inout 参数， 为什么？
  • inout 参数必须在函数的作用域内使用，并且会在函数调用结束后直接修改传递给它的变量的值
  • 逃逸闭包是指闭包在函数返回后仍然可能被调用，生命周期延长
  • 结合以上两点我们知道如果捕获，会导致内存方面的问题

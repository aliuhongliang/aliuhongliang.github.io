网络协议

发表于 2025-01-10 更新于 2025-09-05 分类于 iOS
本文主要介绍 网络协议。

http

  • get post head put delete options

  • get 参数有限制 post没有。get是用来获取资源的，post是用来处理资源的

  • 幂等操作，执行一次和执行多次效果是一样的

  • 状态码 1xx 2xx ：成功 3xx：重定向 4xx：客户端异常 5xx：服务端异常

  • 三次握手

    • 客户端发送syn同步报文到服务端
    • 服务端返回syn 和 ack 报文给客户端
    • 客户端返回ack确认报文给服务端
    • 连接后进行数据传输

  • 四次挥手

    • 客户端发送fin终止报文到服务端
    • 服务端返回 ack 报文给客户端
    • 服务端发送 fin 和 ack 报文给客户端
    • 客户端返回 ack报文给服务端

  • http长久连接

    • 减少tcp连接次数，提升网络请求效率
    • 头部字段有 connect：keep-alive， time 20s ， max 10 等
    • 既然是长久连接，如何判断一个请求已经结束了？
      • 头部字段有，content-length：1024，客户端可以根据接受的字节数是否达到了这个值来判断
      • chuncked，根据空chuncked来判断

  • 抓包原理

    • 利用了http中间人攻击漏洞来实现的。
    • 客户端和服务端之间多了一层，可以理解为代理，中间人可以篡改数据。

    chalse捉包 原理：HTTP/HTTPS 代理【http捉包】 和 中间人攻击【https捉包】

    核心一：HTTP代理 (HTTP Proxy)
    这是最简单也是基础的模式，主要针对未加密的HTTP流量。

    设置代理：你将你的设备（电脑或手机）的网络代理设置指向 Charles 运行的机器IP和端口（默认是 8888）。

    流量转发：此后，你设备上所有的HTTP请求都不会直接发送给目标服务器，而是先被发送到 Charles 这个“代理服务器”上。

    拦截与转发：Charles 接收到请求后，可以对其进行记录、分析甚至修改，然后再代表你的设备将请求转发给真正的目标服务器。

    接收响应：服务器返回的响应同样先回到 Charles，Charles 记录后再返回给你的设备。

    结果：这样，Charles 就成为了所有网络流量的“中转站”，自然就能抓到所有的包。

    核心二：中间人攻击 (MITM) - 用于HTTPS
    HTTPS 的目的是为了安全，防止中间人窃听和篡改。而 Charles 抓取 HTTPS 包的本质就是一个受你控制的、合法的“中间人攻击”。它需要你的主动配合（安装证书）来取得你设备的信任。

    关键步骤：
    安装Charles根证书：

    当你在设备上安装并信任了 Charles 的自定义根证书（CA Certificate） 后，你的设备就意味着：“我完全信任由 Charles 这张证书签发的任何网站证书”。

    这是最关键的一步，是所有HTTPS抓包的前提。

    SSL代理握手过程（配合上图理解）：

    当你的设备（客户端）试图与 https://example.com 建立安全连接时，这个请求首先被发往 Charles。

    Charles 拦截这次握手，并立即动态地伪造一张属于 example.com 的“假”证书。这张假证书由之前安装的 Charles 根证书签名。

    由于你的设备信任 Charles 的根证书，因此它也会信任这张伪造的 example.com 证书。Charles 成功“骗”过了你的设备。

    你的设备会与 Charles 基于这张假证书建立一个独立的HTTPS连接。

    与真实服务器的握手：

    与此同时，Charles 会以客户端的身份，与真实的 https://example.com 服务器建立另一个独立的、真正的HTTPS连接。Charles会验证服务器证书的真实性。

    解密、记录、再加密（中间人工作）：

    现在，Charles 坐在两段安全的HTTPS连接中间：

    一段是：你的设备 ←→ Charles (加密，但Charles可解密)

    另一段是：Charles ←→ 真实服务器 (加密)

    你的设备发送给 example.com 的加密数据，会先被 Charles 用自己的私钥解密（解开第一段连接），从而看到明文内容。Charles 记录下这些明文。

    然后，Charles 再将你的请求内容，用真实服务器的公钥重新加密（建立第二段连接），发送给服务器。

    服务器返回的加密响应，流程相反：Charles 先解密看到明文并记录，然后再用为你设备生成的假证书密钥重新加密，发回给你的设备。

    结果：Charles 成功扮演了“中间人”的角色，对于你的设备和真实服务器来说，它们都以为自己在与对方直接进行安全的加密通信，但实际上它们都是在和 Charles 通信。Charles 得以窥见所有流量的明文内容。

    总结：Charles 抓包原理的核心要点

    机制 针对协议 是否需要安装证书 原理
    HTTP 代理 HTTP 不需要 通过设置系统代理，让所有流量先经过Charles，直接记录明文。
    SSL 代理 (MITM) HTTPS 必须安装并信任 通过受控的“中间人攻击”，动态伪造证书，解密

https和网络安全

  • https和http的区别 https是由http和ssl/tls组成的

    • 客户端 发送 tls版本号，支持的加密算法，随机数C
    • 服务端 返回 使用的加密算法，随机数S，server证书
    • 客户端 验证 server证书，验证公钥
    • 客户端 组装 会话秘钥：随机数C+随机数S+预主密钥
    • 客户端 发送 通过server的公钥对预主密钥加密传输（公钥加密）
    • 服务端 通过私钥对预主密钥解密（私钥加密）
    • 服务端 组装 会话秘钥：随机数C+随机数S+预主密钥
    • 客户端 发送 加密的握手消息
    • 服务端 发送 加密的握手消息
    • 最后两步骤就是验证加密通道是否连接完成
    • 建立完成后，使用对称加密对数据进行加密传输

    HTTPS 工作流程
    HTTPS 会对 URL 的 path、query 参数加密，外部看不到请求的路径和参数。不会加密域名，因为域名在 TLS 握手时就需要暴露。所以：外部能看到你访问了 example.com，但不知道你访问了 example.com/abc?password=123。

    1. 客户端发起请求

    用户在浏览器输入 https://example.com，客户端告诉服务器它要进行 HTTPS 通信。

    客户端会发送一个 ClientHello 消息，其中包含：

    支持的 TLS 版本（如 TLS1.2 / TLS1.3）

    支持的加密套件（RSA/ECDHE/AES 等）

    一个随机数（client random）

    1. 服务器响应

    服务器返回一个 ServerHello 消息，包含：

    确认使用的 TLS 版本和加密套件

    一个随机数（server random）

    同时，服务器会把自己的 数字证书（包含服务器公钥、公钥签名、域名信息等）发给客户端。

    1. 客户端验证证书

    客户端验证服务器证书是否可信：

    检查证书是否过期

    检查证书是否由受信任 CA 签发

    检查证书上的域名是否与访问的域名一致

    验证通过后，进入密钥交换阶段。

    1. 密钥协商（核心部分）

    有两种常见方式：

    (1) RSA 密钥交换（较老）

    客户端生成一个 对称密钥（pre-master secret）。

    用服务器公钥（来自证书）加密该对称密钥并发给服务器。

    服务器用私钥解密得到对称密钥。

    (2) ECDHE 密钥交换（推荐，支持前向保密 PFS）

    客户端和服务器各自产生一对临时密钥对（椭圆曲线 Diffie-Hellman）。

    双方交换公钥，并使用对方公钥和自己私钥计算出相同的 会话密钥。

    这个密钥不会被窃听者推导出来。

    1. 生成会话密钥

    客户端和服务器通过 随机数（client random + server random）+ pre-master secret 共同计算出最终的 会话密钥。

    后续所有通信都使用该会话密钥进行 对称加密。

    1. 握手完成

    双方互相发送一个 Finished 消息，表示握手成功。

    之后的 HTTP 请求/响应就通过 TLS 加密的通道传输。

  • TCP UDP

  • tcp：传输控制协议

    • 特点：面向连接，可靠传输，面向字节流，流量控制，拥塞控制
      • 面向连接：数据传输时，需要先建立连接，结束时，需要断开连接，3次握手，4次挥手。
        为什么不能两次握手，如果第一次syn发送超时的时候，客户端会重传，如果只有2次，这一次就建立连接了，如果第一次超时的syn这时候再传输到服务器，这是可能服务器就理解为再一次连接。
        为什么是4次？客户端和服务端是全双工通讯，两端都可以发送和接受，所以要分别断开。
      • 可靠传输：无差错，不丢失，不重复，按序到达。解答：每次数据传递都有确认机制来保证可靠
      • 面向字节流
      • 流量控制：根据滑动窗口协议实现的。根据报文头中的数据，动态调解发送窗口的大小，来实现流量控制。发送方和接收方都有一个数据缓存，如果接收方来不及处理发送方数据的时候，就是告诉发送方少发一点。同理接受窗口类似
      • 拥塞控制：慢开始避免拥塞、快恢复快重传。

  • udp：用户数据包协议

    • 特点：无连接、不保证可靠传输、对报文不合并也不拆分
      • 功能：复用，分用，差错检测
      • 复用：同一个ip 不同的端口，都可以通过udp传输
      • 分用：数据报通过ip层到udp层，可以根据目的端口分发到对应的端口

  • dns

  • 域名到ip地址的映射，dns解析请求是通过udp数据报并且是明文的。 client 1、向 dns服务器 获取ip 2、向server请求
    查询方式：递归查询（一层一层的问）、迭代查询（其他dns告诉本地dns问谁要ip）

  • dns解析存在的问题？

    • dns劫持：dns是明文传输，就有可能被劫持篡改。dns发送在http连接之前
    • 解析转发：举例：本该是移动的ip结果被转发给了电信的ip
    • 如何解决劫持。httpDNS，使用http协议向dns服务器的80端口进行请求。长连接，使用长连server，客户端和长连server连接，长连server和服务端进行http通讯

  • session cookie

    • 为了解决http无状态的特点，http不知道上一次请求的用户，引入了session和cookie。cookie是用来记录用户状态，用来区分用户，状态存在于客户端，每次请求的时候，服务端生成cookie，通过响应报文传递给客户端，客户端保存cookie，后续的请求客户端需要在请求报文中携带cookie传递给服务端。
